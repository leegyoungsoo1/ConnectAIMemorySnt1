### 🛡️ 쇼핑몰 프로젝트 계획서 보안 설계 점검 결과

**1. [A02:2021-암호화 실패] 인증 토큰 관리 전략 미흡 — CVSS: 5.3 (Medium)**
- **분석:** 기술 스택에 JWT(JSON Web Token) 사용이 명시되어 있습니다. [근거: shopping_mall_project_plan.md] 하지만 토큰의 저장소(LocalStorage vs HttpOnly Cookie)와 만료 전략(Refresh Token 도입 여부)이 구체적이지 않습니다. LocalStorage에 저장할 경우 XSS 공격 시 토큰 탈취 위험이 매우 높습니다. [추측]
- **대응 방안:** Access Token은 메모리에, Refresh Token은 `HttpOnly`, `Secure` 설정이 된 쿠키에 저장하는 전략을 설계에 반영해야 합니다.

**2. [A01:2021-접근 제어 취약점] IDOR(부적절한 직접 객체 참조) 가능성 — CVSS: 7.5 (High)**
- **분석:** 4주차 일정에 '권한 관리(IDOR) 취약점 점검'이 포함되어 있는 점은 긍정적입니다. [근거: shopping_mall_project_plan.md] 하지만 쇼핑몰 특성상 `orderId`나 `userId`를 기반으로 API 요청을 보낼 때, 서버에서 세션 사용자와 요청 리소스의 소유권을 검증하는 로직이 누락될 경우 타인의 주문 정보를 조회/수정할 수 있는 치명적인 취약점이 발생합니다. [추측]
- **대응 방안:** 모든 API 엔드포인트에서 `Resource Owner` 검증 로직을 강제하는 공통 인터셉터 설계를 요청합니다.

**3. [A04:2021-안전하지 않은 설계] 입력값 검증 체계의 구체성 부족 — CVSS: 4.3 (Medium)**
- **분석:** '서버 사이드 이중 검증 체계'를 구축한다고 명시하였습니다. [근거: shopping_mall_project_plan.md] 하지만 구체적으로 어떤 라이브러리(예: class-validator, Joi 등)를 사용하여 화이트리스트 기반의 검증을 수행할 것인지에 대한 정의가 없습니다. [추측]
- **대응 방안:** DTO(Data Transfer Object) 수준에서 엄격한 타입 체크와 길이 제한, 특수문자 필터링 정책을 수립해야 합니다.
